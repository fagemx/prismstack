# 教學框架

> 用途：引導員「怎麼教」的方法論。不是教什麼（那是 methodology-anchors.md），而是怎麼把概念變成用戶能吸收的教學。
> 引導員根據情境選擇框架，不是每次都用同一套。

---

## 框架 1: 情境拆解法（主要框架）

> 適用：情境教學模式。用戶想學一個新能力。

### 步驟

```
1. 確認目標
   用戶想學什麼？翻譯成一句話：「學完之後你能做到 ___」

2. 拆成 2-4 個動手步驟
   不是概念步驟，是操作步驟。每步都有具體的動作：
   - 打開某個檔案
   - 跑某個命令
   - 改某一行
   - 觀察某個結果

3. 每步執行 Do-Then-Explain 迴圈：
   a. 做 — 帶用戶一起操作（讀檔案、改內容、跑命令）
   b. 看 — 指出剛才操作產生了什麼變化
   c. 解 — 用一句話解釋為什麼這樣做（連結到方法論錨點）
   d. 延 — 如果自然，延伸到相關概念（「這也適用在 X」）

4. 收尾
   用一句話總結：「所以 ___ 的核心就是 ___」
```

### 範例

用戶問：「怎麼讓兩個 skill 串起來？」

```
目標：學完之後你能讓 /extract 的產出被 /transform 自動讀到

步驟 1（做）：打開 /extract 的 SKILL.md，找到結尾的 save 邏輯
步驟 1（看）：它存到 ~/.prismstack/projects/slug/extract-*.md
步驟 1（解）：每個 skill 完成時把 artifact 存到固定位置，這就是「輸送帶」

步驟 2（做）：打開 /transform 的 SKILL.md，找到 Phase 0 的 discovery
步驟 2（看）：它搜尋 *-extract-*.md — 剛好對上
步驟 2（解）：下游 skill 用 glob pattern 自動找上游的產出

步驟 3（做）：實際跑一次 /extract，然後看 projects/ 目錄
步驟 3（看）：出現了 nox-main-extract-2026-03-25-1430.md
步驟 3（延）：你的 /review 也用同樣的機制找 /extract 的產出

收尾：artifact flow 的核心就是「固定位置 + 固定命名 + 自動搜尋」
```

## 框架 2: 類比生成法

> 適用：解釋抽象概念時。讓陌生概念變成用戶已知的東西。

### 步驟

```
1. 辨識概念的本質
   這個概念在做什麼？（不是叫什麼名字）

2. 找用戶領域的對應物
   在用戶的工作裡，有什麼東西跟這個概念的本質一樣？
   - 電商：訂單流、庫存管理、物流追蹤
   - 內容：稿件流、審核鏈、發布管線
   - 教育：課程設計、學生進度追蹤、評量回饋
   - 諮詢：案件流、交付里程碑、客戶回饋迴圈

3. 建立映射
   概念的每個部分 ↔ 類比的每個部分
   確保映射是精確的，不是模糊的

4. 用類比解釋
   先說類比，再說概念。不要反過來。
   ✅「你知道電商訂單從下單到出貨怎麼走的嗎？artifact flow 就是那個」
   ❌「artifact flow 是 skill 之間的檔案傳遞，就像電商訂單」

5. 指出類比的邊界
   類比哪裡會失效？主動說出來，避免過度延伸。
   「不過跟訂單不同的是，artifact 不會被『取消』— 只會被新版取代」
```

### 類比品質檢查

| 好的類比 | 壞的類比 |
|---------|---------|
| 用戶秒懂（不需要再解釋類比本身） | 類比比原概念更複雜 |
| 映射精確（概念的每部分都有對應） | 只有表面像（「就像蓋房子」但沒說哪裡像） |
| 指出邊界（哪裡類比失效） | 暗示完全等價 |

## 框架 3: 對比消歧法

> 適用：用戶混淆兩個類似概念時。

### 步驟

```
1. 並排對比
   | 面向 | 概念 A | 概念 B |
   |------|--------|--------|
   | 做什麼 | ... | ... |
   | 產出什麼 | ... | ... |
   | 什麼時候用 | ... | ... |

2. 一句話區分
   「A 是 ___，B 是 ___。最大的差別在 ___。」

3. 用用戶的 skill 舉例
   「你的 /X 是 A 類型，/Y 是 B 類型 — 看它們的產出就知道」

4. 邊界案例
   「在 ___ 情況下兩個很像，但判斷標準是 ___」
```

### 常見混淆對

| 混淆 | 區分 |
|------|------|
| Review vs Tester | Review 判品質（好不好），Tester 找問題（哪會壞） |
| Bridge vs Production | Bridge 轉格式（輸入≠輸出格式），Production 從無到有 |
| STOP gate vs Phase 切換 | STOP 是問用戶，Phase 是流程進度 |
| Skill vs Reference | Skill 改行為模式，Reference 是背景知識 |
| Artifact vs Output | Artifact 有固定格式命名可被下游搜尋，Output 只是結果 |

## 框架 4: 升級路徑法

> 適用：用戶已經會基礎，想進階。

### 步驟

```
1. 確認用戶目前的水平
   能做到什麼？（不問「你覺得你在什麼程度」— 從行為判斷）

2. 展示下一級
   你目前在 Level N，下一級 Level N+1 是 ___
   差別在於：Level N 手動做 ___，Level N+1 自動做 ___

3. 帶做一個升級動作
   不是講「你可以做」，是「來，我們現在做一個」

4. 確認效果
   做完後，指出變化：「你看，現在這步不需要手動了」
```

### 常見升級路徑

```
手動跑單一 skill
  → 理解 artifact flow，讓 skill 自動串接
    → 用 /skill-gen 加新 skill 到 pipeline
      → 用 /tool-builder 把重複操作包成 helper
        → 用 auto mode 跑整條 pipeline
          → 用 /domain-upgrade 持續優化
```

## 框架 5: 反面教學法

> 適用：強調某個原則的重要性時。先看「做錯會怎樣」，再看「正確做法」。

### 步驟

```
1. 展示反面案例
   「如果不做 X，會發生 Y」
   用用戶的 stack 舉例（如果可能）或用常見錯誤

2. 讓用戶感受後果
   不是說「這樣不好」— 是展示具體的壞結果
   「看，因為沒有 scoring formula，AI 給了 9/10 但其實品質只有 6/10」

3. 對比正確做法
   「現在加上 scoring formula，看結果變什麼」

4. 提煉原則
   「所以 ___ 的原則就是為了防止 ___」
```

## 選擇哪個框架

| 用戶狀態 | 主要框架 | 輔助框架 |
|---------|---------|---------|
| 想學新能力 | 情境拆解法 | 類比生成法 |
| 聽不懂概念 | 類比生成法 | 對比消歧法 |
| 搞混兩個概念 | 對比消歧法 | — |
| 會基礎想進階 | 升級路徑法 | 情境拆解法 |
| 不理解為什麼要遵循某規則 | 反面教學法 | 情境拆解法 |

| 反覆修卻修不好 | 根因追溯法 | 反面教學法 |
| 跳過步驟、找藉口 | 合理化識別法 | 反面教學法 |
| 要不要寫腳本、怎麼呼叫工具 | 腳本層決策法 | 對比消歧法 |

**混用是正常的。** 一次教學中可能用到 2-3 個框架。例如：用情境拆解法帶操作，中途用類比生成法解釋抽象概念，最後用升級路徑法指出下一步。

## 框架 6: 根因追溯法

> 來源：autoresearch debug workflow + superpowers systematic debugging
> 適用：用戶的 skill 有問題但改了又冒出新問題；pipeline 某處反覆失敗。

### 步驟

```
1. 鎖定症狀
   用戶看到的問題是什麼？（不是用戶覺得的原因，是觀察到的現象）
   「/review 的評分不準」→ 這是症狀
   「scoring formula 權重不對」→ 這是用戶的假設，先放一邊

2. 向後追蹤（5 Whys）
   從症狀出發，每一層問「為什麼」：
   - Why 1: 為什麼評分不準？→ 權重是通識的
   - Why 2: 為什麼是通識的？→ /domain-build 沒有讀 benchmarks
   - Why 3: 為什麼沒有 benchmarks？→ /domain-plan 沒問
   - Why 4: 為什麼沒問？→ 這是根因
   繼續追直到找到「可以修的源頭」

3. 驗證根因
   修根因，看症狀是否消失。
   如果症狀還在 → 不是真的根因，繼續追。

4. 教學連結
   教用戶追溯的思路，不只是給答案：
   「你看，問題表面在 /review，但根源在 /domain-plan。
   以後遇到類似狀況，先問『這個東西是誰產出的？那個產出者的輸入又來自哪裡？』」
```

### 架構升級門

```
修第 1 次 → 還有問題
修第 2 次 → 問題換地方出現
修第 3 次 → STOP.

不要嘗試第 4 次修復。
這不是修復問題，是設計問題。

帶用戶退一步：
- 這個 skill 的職責是不是太大了？
- 上下游的 artifact 合約對嗎？
- 這個 skill 應該存在嗎？
```

### 追溯工具

| 工具 | 用法 |
|------|------|
| artifact 的 Supersedes chain | 看這個 artifact 怎麼演變的 |
| decision-log.jsonl | 看 skill 的決策歷史 |
| git log / git diff | 什麼時候改了什麼 |
| Phase 0 的 discovery 結果 | 上游 artifact 是否存在、格式是否正確 |

## 框架 7: 合理化識別法

> 來源：superpowers rationalization tables + red flags
> 適用：用戶（或 AI）在找藉口跳過步驟、偷工減料時。

### 原理

人和 AI 都會合理化 — 把「不想做」包裝成「不需要做」。
引導員的工作不是阻止用戶，而是幫用戶**看見自己在合理化**。

### 步驟

```
1. 辨識合理化信號
   用戶或 AI 說的話是否符合以下模式：
   - 「這個太簡單不需要 X」（X = 測試/review/STOP gate）
   - 「先做完再說」（跳過規劃直接動手）
   - 「這次例外」（繞過品質標準）
   - 「我知道該怎麼做」（跳過學習環節）
   - 「以後再補」（永遠的以後）

2. 不直接否定
   ❌「你不能跳過這步」
   ✅「可以。但先讓我們看看跳過會怎樣」

3. 用反面教學展示後果
   快速展示：跳過這步 → 具體會發生什麼壞事
   用用戶自己的 stack 舉例（不是假設性的恐嚇）

4. 讓用戶自己決定
   展示後果後，用戶仍然可以選擇跳過。
   引導員記錄這個選擇，下次遇到相關問題時可以回溯。
```

### 常見合理化對照表

| 合理化 | 表面邏輯 | 實際風險 |
|--------|---------|---------|
| 「這個 skill 太簡單不需要 gotchas」 | 簡單的東西不會出錯 | 簡單 skill 被 AI 跑得最頻繁，沒有 gotchas = 每次都可能犯同樣的錯 |
| 「先跑起來再加 scoring」 | 先有再好 | 沒有 scoring 的 skill 跑完你無法判斷結果好不好，等於白跑 |
| 「STOP gate 太多太慢」 | 效率優先 | 新 stack 沒有 STOP gate = AI 一路跑到底但可能從第 2 步就錯了 |
| 「我知道 artifact flow 所以不用教」 | 已經會了 | 「知道」和「能操作」不同。能不能現在用你的 skill 示範一次？ |
| 「先把所有 skill 寫完再測」 | 批次效率 | 10 個沒測的 skill vs 3 個測過的 skill — 後者更有用 |
| 「這次不用 anti-sycophancy」 | 不是每個 skill 都需要 | 沒有 anti-sycophancy 的 review skill = 永遠給好評的審查官 |

### 教學原則

- **目標不是服從，是理解** — 用戶理解風險後選擇跳過 ≠ 合理化。用戶沒看到風險就跳過 = 合理化。
- **紀錄而不阻止** — 引導員記錄用戶的選擇，不強制。但下次遇到後果時可以回溯：「你還記得上次選擇跳過 X 嗎？這就是後果」。
- **自己也檢查** — 引導員自己也可能合理化（「這個用戶很厲害不需要教基礎」）。用紅旗清單自檢。

## 框架 8: 腳本層決策法

> 來源：jimeng-auto 實戰經驗 — 18 個 skill 中 16 個純 prompt、2 個用腳本
> 適用：用戶在建 skill 時問「要不要寫腳本」「怎麼呼叫 API」「要不要加 scripts/」

### 步驟

```
1. 盤點用戶的專案工具
   用戶的專案有哪些可以直接呼叫的東西？
   - CLI 工具（node xxx.js、python xxx.py）
   - API endpoint
   - 現成的 library

2. 走判定樹
   帶用戶走一遍三層判定樹（methodology-anchors.md 錨點 14）：
   - 需要外部工具嗎？
   - 專案已有 CLI 嗎？
   - 會被多次/多 skill 共用嗎？
   - 每次需要 SKILL.md 判斷結果嗎？

3. 用用戶的 skill 做分類
   掃描用戶現有的 skill，逐個判斷：
   「你的 /extract 用到 Python script → Layer 2」
   「你的 /review 是純 prompt → Layer 1」
   「你的 /batch-produce 跑 100 個任務 → Layer 3」

4. 如果需要寫腳本（Layer 2 或 3）→ 帶做一個
   a. 定義 CLI 介面（--task、--input、--output）
   b. 定義 exit code 規範
   c. 寫最小可用版本
   d. 在 SKILL.md 裡寫調用指令
   e. 跑一次驗證

5. 如果不需要腳本（Layer 1）→ 確認直接呼叫模式
   a. 找到專案的 CLI 命令
   b. 在 SKILL.md 的對應 Phase 裡寫呼叫指令
   c. 跑一次驗證
```

### 常見教學場景

| 場景 | 用戶說 | 引導 |
|------|--------|------|
| 過度包裝 | 「我要為每個 API 寫 wrapper」 | 「專案已經有 CLI 了嗎？有就直接用。wrapper 只在 CLI 不存在時才寫」 |
| 該寫不寫 | 「我把 API 呼叫全寫在 SKILL.md 的 inline bash 裡」 | 「超過 50 行就該移到 scripts/。而且如果其他 skill 也要用，封裝成工具更划算」 |
| Layer 混淆 | 「我的 API 客戶端跑完自己做決定」 | 「決策留在 SKILL.md。腳本只做執行和回傳結果。Layer 2 的重點是 SKILL.md 看結果再決定」 |

### 教學品質確認

教完後問用戶：
- 「你的這個 skill 是 Layer 幾？為什麼？」
- 「如果有另一個 skill 也需要呼叫同一個 API，你會怎麼處理？」（答案：共用 Layer 2 腳本）
- 「什麼時候你會把 inline bash 移到 scripts/？」（答案：>50 行，或被多次/多 skill 使用）

## 教學品質自檢

每段教學完成後，引導員自問：

| 檢查 | 通過 | 沒通過怎麼辦 |
|------|------|-------------|
| 用了用戶自己的 skill/artifact 當例子嗎？ | ✅ | 重做，找用戶 stack 裡的具體對象 |
| 用戶動手操作過了嗎？ | ✅ | 補一個動手步驟 |
| 解釋用了領域語言而不是方法論術語嗎？ | ✅ | 重新用類比翻譯 |
| 有延伸到其他可套用的場景嗎？ | ✅ | 主動延伸一個（但只一個） |
| 用戶能自己判斷了嗎（不只是記住規則）？ | ✅ | 出一道 forcing question 確認 |
