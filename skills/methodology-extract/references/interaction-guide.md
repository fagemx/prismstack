# Interaction Guide — 碰撞式互動規則

## 核心原則
1. **不是問卷** — 不要一題一題問。等用戶帶來東西。
2. **帶著 A 看 B** — 永遠先確認 A 是什麼，再讀 B。
3. **提取不是摘要** — 摘要是「B 裡有什麼」，提取是「B 裡什麼對 A 有用」。
4. **呈現解讀讓用戶確認** — 不是問用戶答案，是給他你的解讀讓他修正。

## 什麼時候追問
- 用戶的修正暗示了更深的原則 -> 追問具體
- 提取到的 pattern 跟之前的 collision 矛盾 -> 確認哪個對
- 用戶說「不是這樣」但沒說「是怎樣」 -> 追問

## 什麼時候不追問
- 用戶確認「對」 -> 記錄，不追加問題
- 用戶給了新材料 -> 去讀，不先問
- 用戶看起來想繼續做別的事 -> 存好 collision，結束

## 怎麼偵測用戶輸入的性質
不問「這是什麼類型」，自己判斷：

| 信號 | 性質 |
|------|------|
| 出現「應該」「好的 X 要有」「原則是」 | 原理原則 |
| 出現「先做 X 再做 Y」「流程是」「步驟」 | 操作流程 |
| 出現工具名、參數、格式、API | 技術工藝 |
| 出現「A 部門跟 B 部門」「共用」「分開」 | 架構整合 |
| 混合多種 -> 拆開，每種分別處理 |

## 什麼時候建議合成
- 5+ collisions 沒有合成過 -> 「你已經有 N 次碰撞了，要我整理成方法論嗎？」
- 用戶問「目前方法論是什麼」 -> 立刻合成
- Skill completion 時 -> 自動檢查要不要合成

## Escape hatches
- 用戶給了完整 spec -> 不做碰撞，直接合成成 methodology（標注 confidence: high）
- 用戶只是閒聊不是帶材料 -> 不強制碰撞，正常對話
- 用戶明確說「把這個轉成 skill」-> redirect to /source-convert
