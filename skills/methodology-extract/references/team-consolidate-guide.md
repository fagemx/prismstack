# 團隊整合指南

> 用途：/methodology-extract 的 team consolidation mode。
> 場景：多人各有自己的 prompt / SOP / 清單 / 經驗，要整合成一份團隊共用的方法論。

## 核心原則

**不是合併文件，是帶著同一個 A 問題看多份材料。**

每份材料是一個獨立碰撞。碰撞之間做比對。最後合成。

## 流程

### Step 1: 收集來源

問用戶：「團隊裡有哪些人的材料可以用？」

列成清單：
```
| # | 人 | 材料 | 格式 | 預估品質 |
|---|---|------|------|---------|
| 1 | 小明 | 審查 prompt | 500 行 txt | 詳細但可能有個人偏好 |
| 2 | 小華 | 審查標準 | 3 句話 | 精煉但可能不完整 |
| 3 | 主管 | SOP | Word 文件 | 有流程但缺評分 |
| 4 | Jinx | 檢查清單 | Excel | 技術面強 |
```

### Step 2: 定 A 問題

跟用戶確認：整合的目標是什麼？

```
好的 A：「我們團隊的素材審查標準應該是什麼」
好的 A：「新人來了要怎麼教他審素材」
不好的 A：「幫我整理大家的東西」（太模糊）
```

A 必須明確到足以判斷「B 裡什麼有用什麼沒用」。

### Step 3: 逐一碰撞

按照收集順序（或用戶指定順序）逐一碰撞。

**第一份材料 = 基線（baseline）。** 提取完就是目前的累積。

**第二份起 = 跟累積比對。** 每次碰撞後分四類：

| 分類 | 判斷標準 | 處理方式 |
|------|---------|---------|
| **重疊** | 新材料裡的東西，之前已經有了 | 標記 confidence: high（多人共識） |
| **衝突** | 新材料跟之前的說法矛盾 | 記錄兩邊，標記待解決 |
| **新增** | 新材料裡有但之前完全沒有的 | 加入累積，標記來源 |
| **gap** | 到目前為止沒有任何人提到的面向 | 加入 open questions |

### Step 4: 衝突解決

**不要自己解決衝突。呈現給用戶。**

衝突超過 3 個或所有碰撞結束時，批量呈現：

```
整合過程中發現 N 個衝突：

1. [衝突主題]
   [人 A] 說：[觀點]
   [人 B] 說：[觀點]
   → 選項

2. ...
```

每個衝突提供選項（不超過 3 個），附帶 RECOMMENDATION。

**衝突解決規則：**
- 多人一致 > 一人獨持
- 有數據支撐 > 純經驗
- 操作層面的衝突 → 用戶決定（沒有對錯）
- 原則層面的衝突 → 追問「為什麼」，找到更深層的共識

### Step 5: 合成

所有碰撞 + 衝突解決完後，合成一份 Methodology Note。

使用 `references/methodology-template.md` 格式。額外標注：

```
## 來源對照

| 內容 | 來自誰 | Confidence |
|------|--------|-----------|
| 構圖審查維度 | 小明 + 小華（共識） | high |
| CTA 可讀性檢查 | 小華（獨有） | medium |
| 解析度規格 | Jinx（技術面） | high |
| 審查流程順序 | 主管 SOP | high |
| 合規檢查 | 無人提及 → gap | — |
```

## 比對的具體判斷

### 怎麼判斷「重疊」
- 兩人用不同的詞說同一件事 → 重疊（例：「構圖要好」vs「排版要合理」）
- 不確定是不是同一件事 → 問用戶

### 怎麼判斷「衝突」
- 兩人對同一個面向給出矛盾的建議 → 衝突
- 不是所有不同都是衝突：小明多提了一個維度，小華沒提 ≠ 衝突，= 新增

### 怎麼判斷「gap」
- 用五件事（經驗、操作、目的、成果、驗證）當 checklist
- 如果某件事所有人都沒提到 → gap
- Gap 不一定要補：可能這個領域確實不需要
